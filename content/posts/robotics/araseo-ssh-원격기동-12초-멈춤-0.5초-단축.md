## 증상

ARASEO 관제 대시보드에서 로봇을 SSH로 원격 기동하는 버튼을 누르면, **약 12초 동안 화면 전체가 응답하지 않았습니다.** 그동안 다른 관제 기능까지 모두 막혀서, 운영자 입장에서는 시스템이 죽은 것처럼 보였습니다.

12초가 지난 뒤에는 로봇이 정상적으로 동작하기 시작하였고, 그 이후로는 평상시처럼 멀쩡히 돌았습니다. 즉 "기동은 되는데, 기동 명령을 보낸 화면이 너무 오래 멈춰 있다"는 문제였습니다.

## 추적

처음에는 네트워크 지연이나 로봇 측 부팅 시간을 의심하였으나, 같은 명령을 터미널에서 직접 SSH로 보내면 즉시 프롬프트가 돌아왔습니다. 즉 SSH 자체가 느린 게 아니라, **관제 측에서 SSH를 호출하는 방식**이 멈춤을 만들고 있었습니다.

호출부를 들여다보니 대략 이런 형태였습니다.

- 관제 서버(FastAPI)에서 `ssh user@robot "ros2 launch ..."` 형태로 명령을 실행
- 실행한 프로세스의 종료를 기다리며 stdout/stderr를 함께 받음

문제는 `ros2 launch`가 **포그라운드 프로세스**로 떠 있는 동안에는 절대 종료되지 않는다는 점이었습니다. 즉 SSH 입장에서는 "원격 명령이 아직 끝나지 않았다"고 보고 계속 기다리고, 관제 측은 그 SSH 호출이 끝날 때까지 다음 일을 못 한 채 멈춰 있었습니다.

12초라는 숫자는 SSH의 ServerAliveInterval·TCP 타임아웃 등이 조합되어 떨어진 결과로 보였습니다. 결국 SSH는 어떻게든 한 번 끊겼지만, **그 한 번이 끊길 때까지의 12초 동안 화면이 잠겨 있던 것**이 본질이었습니다.

## 접근

해결의 핵심은 **원격 프로세스의 입출력을 SSH 세션에서 떼어내는 것**이었습니다. 입출력이 떼어지면 SSH는 명령을 보낸 즉시 자기 할 일을 끝내고 연결을 종료할 수 있고, 원격 프로세스는 그 뒤에도 살아남아 동작합니다.

적용한 조치는 세 가지였습니다.

1. **입출력 리디렉션** — 원격에서 띄울 명령의 stdin/stdout/stderr를 모두 파일이나 `/dev/null`로 돌려, SSH 채널을 통해 흘러나오는 입출력이 없도록 만들었습니다.
2. **독립 세션으로 분리** — `nohup` + `setsid`(또는 동등한 옵션)로 원격 프로세스를 부모(SSH)에서 떼어, 부모가 죽어도 원격 프로세스는 계속 살도록 하였습니다.
3. **백그라운드 실행 + 즉시 반환** — 원격 측 셸이 명령을 백그라운드로 띄우자마자 셸 자체가 종료되도록 하여, SSH가 즉시 반환되도록 하였습니다.

대략 다음과 같은 형태로 정리되었습니다.

```bash
ssh user@robot "nohup setsid <원격_명령> </dev/null >/tmp/robot.log 2>&1 &"
```

관제 측 FastAPI 핸들러에서도 이 호출을 **타임아웃이 짧은 짧은 명령**으로 다루고, 별도의 헬스체크 토픽으로 기동 결과를 확인하도록 책임을 분리하였습니다.

## 결과

| 항목 | 변경 전 | 변경 후 |
|---|---:|---:|
| 관제 화면 응답 정지 시간 | 약 **12초** | 약 **0.5초** |
| 관제 화면 다른 기능 사용 가능 여부 | 12초 동안 불가 | 즉시 가능 |
| 기동 결과 확인 방식 | SSH 반환 대기 | 별도 헬스체크 토픽 |

체감으로는 "버튼을 누르면 즉시 반응한다"로 바뀌었습니다. 운영자 입장에서는 동일 시간 안에 여러 로봇을 연속으로 기동할 수 있게 되었고, 한 로봇 기동 때문에 다른 로봇의 상황을 못 보는 사각지대가 사라졌습니다.

## 배운 점

- 원격 프로세스를 띄우는 호출에서 **호출자가 멈추는 시간** = 원격 프로세스 실행 시간이 아니라, **호출 채널이 정리되기까지 걸리는 시간**이라는 점이 핵심이었습니다.
- "SSH로 백그라운드 실행을 하려면 입출력을 끊어야 한다"는 것은 셸 운영자에게는 익숙한 패턴이지만, 웹 핸들러 코드 안에 SSH 호출을 직접 박아두면 그 감각이 잘 작동하지 않습니다. 핸들러 = 즉시 반환, 결과 확인은 별도 채널이라는 책임 분리를 처음부터 두는 편이 안전합니다.
- 단일 호출을 짧게 줄였다고 끝이 아니라, **결과 확인을 어떤 채널로 받을지**까지 같이 설계해야 운영 사고를 막을 수 있습니다.

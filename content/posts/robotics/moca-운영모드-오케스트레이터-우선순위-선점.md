## 배경

MOCA는 카페에서 다음 다섯 가지 모드를 오가며 동작합니다.

- **서빙(Serving)** — 주문 받은 자리로 음료를 가져다 줌
- **순회(Patrol)** — 매장을 천천히 돌며 공석·상태를 살핌
- **안내(Guiding)** — 손님을 특정 자리로 안내
- **모객(Soliciting)** — 손님에게 다가가 행사·메뉴를 권유
- **대기(Idle)** — 지정된 충전·대기 위치로 복귀

운영자는 웹 대시보드에서 이 모드들을 원격으로 켜고 끌 수 있어야 합니다. 그런데 모드 전환을 단순히 "현재 모드를 새 모드로 바꾼다"로만 두면 두 가지 사고가 곧바로 일어났습니다.

1. **모드 충돌** — 서빙 중인 로봇에게 모객 명령이 들어오면, 음료를 든 채로 다른 손님에게 다가가다 음료를 흘릴 수 있었습니다.
2. **모드 잔류** — 안내 작업이 끝났는데도 안내 모드 상태로 머물러, 다음 주문이나 순회 트리거를 받지 못한 채 가만히 멈춰 있었습니다.

운영자가 매번 수동으로 정리해 줘야 한다면 한 명이 한 대를 보기에도 빠듯해집니다. 모드 전환·복귀를 **운영 서버(OpServer)**가 자동으로 통제하도록 설계가 필요했습니다.

## 접근

### 1) 모드별 우선순위 — 높은 작업이 낮은 작업을 선점

먼저 다섯 모드에 **우선순위**를 부여하였습니다. 카페 운영의 실제 중요도를 기준으로 두었습니다.

| 우선순위 | 모드 | 사유 |
|:---:|---|---|
| 높음 | **서빙** | 손님이 이미 주문한 상태, 음료가 식거나 흘리면 안 됨 |
| ↑ | **안내** | 손님이 자리 안내를 기다리는 상태 |
| ↓ | **순회** | 능동적 매장 살핌, 즉시성 중요하지 않음 |
| ↓ | **모객** | 손님 호의 필요, 즉시성 낮음 |
| 낮음 | **대기** | 아무 일도 없을 때만 |

OpServer의 모드 오케스트레이터는 새 모드 요청이 들어오면 **현재 모드의 우선순위와 비교**합니다.

- 새 모드가 **더 높음** → 현재 모드를 우아하게 중단(graceful preempt)시키고 새 모드로 전환.
- 새 모드가 **같거나 낮음** → 큐에 적재해 두고 현재 모드 완료 후 검토.

서빙 중에 모객 명령이 들어오면 거절되어, 위에서 언급한 "음료 흘림" 사고가 구조적으로 막힙니다.

### 2) 전환은 비동기 — 응답이 없으면 재시도

모드 전환 자체도 즉시 완료되지 않습니다. 현재 작업의 안전 종료(예: 서빙 중이면 음료를 트레이 안전 위치로 정렬), 새 작업의 초기화(예: 안내 모드면 목표 좌석 확인) 등이 몇 초 단위로 일어납니다.

오케스트레이터는 전환 요청을 **비동기**로 처리합니다. 요청 즉시 "수락/거절"만 반환하고, 실제 전환은 백그라운드에서 진행됩니다. 진행 상태는 WebSocket으로 대시보드에 푸시되어, 운영자가 진행률·실패 여부를 실시간으로 봅니다.

전환 중 일정 시간 안에 응답(상태 변화·완료 신호)이 없으면 **재시도**합니다. 재시도 횟수와 백오프 간격을 모드별로 둘 수 있게 두어, 비교적 가벼운 모드(대기)는 빠르게 재시도하고 무거운 모드(서빙)는 천천히 재시도하도록 하였습니다. 재시도 한계를 초과하면 **실패 알림 + 대기 모드로 안전 복귀**합니다.

### 3) 작업 완료 감지 → 자동 대기 복귀 → 일정 시간 후 자동 순회

모드 잔류 문제는 **완료 감지와 자동 복귀**로 해결하였습니다.

- 순회·안내 같은 시간이 정해진 작업은 BT(BehaviorTree)에서 완료 이벤트를 발행합니다.
- 오케스트레이터는 이 이벤트를 받으면 **자동으로 대기 모드로 복귀**시킵니다. 운영자의 별도 명령이 필요하지 않습니다.
- 대기 상태가 일정 시간(예: 5분) 지속되면 **자동으로 순회 모드를 시작**합니다. 매장이 비는 시간에도 로봇이 능동적으로 매장을 살피게 됩니다.

대기 ↔ 순회 사이의 자동 전환은 우선순위가 같거나 낮은 모드들끼리의 전환이기 때문에, 서빙·안내가 들어오면 즉시 선점되어 우선순위 규칙이 깨지지 않습니다.

## 결과

| 상황 | 변경 전 | 변경 후 |
|---|---|---|
| 서빙 중 모객 요청 | 충돌·음료 흘림 위험 | 우선순위 규칙으로 거절 |
| 안내 완료 후 | 안내 모드 잔류 | 대기 모드로 자동 복귀 |
| 매장 비는 시간 | 운영자가 수동으로 순회 트리거 | 일정 시간 후 자동 순회 |
| 전환 중 응답 없음 | 모드가 어디 있는지 모름 | 재시도 → 실패 시 대기 안전 복귀 |

운영자 한 명이 동시에 보는 로봇 수를 늘릴 수 있게 되었고, "지금 이 로봇은 어떤 상태지?"를 매번 확인하지 않아도 시스템이 안전하게 다음 단계를 진행하였습니다.

## 배운 점

- 다중 모드 시스템의 핵심은 **모드 자체보다 모드 전환을 관리하는 단일 통제점**입니다. 모드 전환 로직을 각 모드 안에 흩어 두지 말고, 오케스트레이터 한 곳에 모아두는 편이 디버깅·확장 모두 유리하였습니다.
- 우선순위는 정책이지 코드 상수가 아닙니다. 운영 중에 "안내가 모객보다 우선" 같은 정책 변화가 생기면, 오케스트레이터 한 곳의 우선순위 표만 바꾸면 됩니다.
- **자동 복귀**가 운영 시스템의 신뢰도를 가장 크게 올려준 변경이었습니다. "끝났는데 안 돌아오는" 상태가 한 번 누적되면 운영자가 시스템을 신뢰하지 않게 되는데, 이걸 시스템이 알아서 처리하면 운영자가 마음 놓고 다른 일을 합니다.

# Obsidian은 Claude Code의 '기억'이 되어줄까 — 착각과 실제

Obsidian에 지식을 쌓고 Claude Code를 붙이면 "AI가 내 모든 걸 기억하는 세컨드 브레인"이 된다 — 이런 글이 많습니다. 저도 그 말을 믿고 직접 만들어봤습니다. 노트북에 흩어진 프로젝트 문서를 전부 컴파일해서 **위키 노트 168개, 링크 1,007개**짜리 Obsidian vault(LLM Wiki 패턴)를 구축했죠.

만들고 나서 든 질문이 이 글의 주제입니다. **"이게 진짜 Claude가 '기억'하는 건가?"**

결론부터 말하면 — **아닙니다. 적어도 흔히 생각하는 의미로는요.** 대신 더 정확한 표현이 있습니다.

---

## Claude Code의 '기억'은 사실 3개의 다른 층입니다

| 층 | 자동으로 불러와지나? | 실제 역할 |
|---|---|---|
| **memory 시스템** (`~/.claude/.../memory/`) | ✅ 세션 시작 시 컨텍스트에 주입 | 이게 진짜 "Claude가 기억"하는 것. 단 작음 |
| **CLAUDE.md** (작업 디렉터리 루트) | ✅ 그 폴더에서 일할 때만 | 그 프로젝트의 규칙·맥락을 "기억" |
| **Obsidian vault 노트들** | ❌ 안 됨 | Claude가 **직접 읽거나 grep 해야만** 알게 됨. 평소엔 컨텍스트에 없음 |

핵심은 마지막 줄입니다. **vault의 168개 노트는 Claude의 머릿속에 들어있지 않습니다.** Claude(모델)는 영구 기억이 없어요. 매 세션 백지로 시작합니다. 영구적인 건 **디스크 위의 파일**뿐이고, 그 파일은 누가 읽어줘야 의미가 생깁니다.

---

## 그래프 뷰는 Claude에게 거의 쓸모가 없습니다

이건 좀 의외였습니다. Obsidian의 상징인 그 멋진 **그래프 뷰 — Claude는 못 봅니다.** 시각적으로 노드를 따라가지도, 그래프 구조를 활용하지도 않아요. 그래프는 **사람(나)이 지식을 탐색하고 사고하는 도구**입니다.

만약 "Claude가 기억하게 하려고" 그래프를 정성껏 다듬고 있었다면, 그건 빗나간 노력입니다. Claude 입장에서 실제로 도움이 되는 건 따로 있습니다:

- **`[[위키링크]]`** — 약간 도움. Claude가 `grep '[[X]]'` 로 관련 노트를 찾아 따라 읽는 **텍스트 인덱스** 역할은 합니다.
- **atomic하게 쪼갠 작은 노트** — 검색 정밀도에 유리. 거대한 한 파일보다 컨텍스트 낭비가 적습니다.
- **MOC(Map of Content) 같은 진입점 문서** — Claude가 "여기부터 보면 되는구나"를 잡는 데 유용.

즉 Claude에게 vault는 **"grep하고 읽을 수 있는 로컬 문서고"** 입니다. 잘 만든 레퍼런스 문서지, 기억이 아닙니다.

---

## 그래서 "기억"이 아니라 "포인터 + 검색"입니다

vault가 다른 프로젝트 작업 중에 실제로 도움이 되려면, 한 가지가 빠져 있습니다 — **연결**.

vault는 `~/nextbrain`에 있는데 실제 코드는 `~/dev_ws/...`에 있습니다. **다른 폴더에서 Claude Code를 켜면 vault의 존재 자체를 모릅니다.** 그래서 각 프로젝트의 `CLAUDE.md`(자동 로드되는 파일)에 포인터 한 줄을 박아둬야 합니다:

```markdown
# Knowledge Center
개인 지식센터가 ~/nextbrain 에 있다 (Obsidian LLM Wiki).
배경지식이 필요하면 wiki/_index/MOC.md 를 먼저 읽고,
grep -rl "<키워드>" ~/nextbrain/wiki 로 관련 노트를 찾아 읽는다.
```

비유하자면 — 책을 통째로 외우게 하는 게 아니라, **책상에 "매뉴얼은 3번 서랍" 포스트잇을 붙이는 것**입니다. 앉을 때마다 포스트잇을 보고 서랍을 열 수 있지만, 매뉴얼이 머릿속에 있는 건 아니고, 매번 새로 펼쳐 읽습니다. (더 무겁게 가려면 Obsidian/filesystem MCP로 vault를 도구처럼 질의하게 만들 수도 있습니다.)

| 영구적인 것 ✅ | 영구적이지 않은 것 ❌ |
|---|---|
| 포인터 한 줄 (자동 로드) | 168개 노트 내용 (매번 새로 읽어야, 토큰 듦) |
| vault 파일들 (디스크) | Claude가 "이미 안다"는 상태 |
| memory 파일 (단, 해당 프로젝트 폴더에서만 자동) | 다른 폴더로 자동으로 따라오지 않음 |

---

## 솔직한 한계 세 가지

만들어보고 알게 된, 잘 안 적히는 부분들입니다.

1. **휘발성** — "이번 세션의 Claude"가 내 프로젝트를 다 이해해도, **다음 세션의 Claude는 그걸 기억하지 못합니다.** memory 파일 + vault를 다시 읽어야 복원됩니다.
2. **읽어도 틀릴 수 있음** — 컴파일된 노트에 오류가 있으면 **틀린 걸 읽고 틀린 걸 "기억"합니다.** 저는 커버리지를 검증보다 우선했기 때문에, 세부 수치는 아직 미검증입니다. "compile once, query forever"의 그림자죠.
3. **의미 검색이 아님** — Claude의 vault 탐색은 임베딩 기반 시맨틱 검색이 아니라 **키워드 grep + 파일명 매칭**입니다. 노트 제목·태그가 검색어와 안 맞으면 못 찾습니다. 그래서 네이밍이 생각보다 중요합니다.

---

## 그럼 의미가 없나? — 아니요, 단지 정확히 알고 쓰자

오해를 걷어내면 이렇게 정리됩니다.

- **사람의 기억·정리·포트폴리오 도구로는** Obsidian(그래프 포함)이 충분히 가치 있습니다.
- **Claude Code의 "기억"으로는**, 그래프는 무관하고, vault는 **"CLAUDE.md 포인터 + 검색 친화적 atomic 노트"** 라는 두 가지를 갖췄을 때 비로소 작동합니다.
- 그리고 어느 쪽이든, **내용 검증**을 안 하면 잘 정리된 오류를 영원히 참조하게 됩니다.

"AI가 내 모든 걸 기억한다"는 마케팅 문장보다, **"AI가 내가 정리해둔 곳을 안정적으로 찾아본다"** 가 실제에 가깝습니다. 덜 멋있지만, 이렇게 알고 설계하면 실제로 쓸 만한 시스템이 됩니다.
